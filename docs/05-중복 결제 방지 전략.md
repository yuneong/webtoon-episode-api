## 중복 결제 방지 전략

### 선택한 방식
**비관적 락(Pessimistic Lock) + DB Unique Key + Idempotency Key** 3중 방어 구조로 설계했습니다.


### 동시 요청 시 중복 결제 방지 흐름
동일 사용자가 동일 회차를 동시에 여러 번 요청하는 상황을 가정합니다.
```text
1차 방어 — Idempotency Key
클라이언트가 요청마다 고유한 UUID를 생성해 헤더로 전달합니다.
purchases 테이블의 idempotency_key 단일 유니크 인덱스로 동일 키의 중복 INSERT를 DB 레벨에서 차단합니다.

2차 방어 — 비관적 락 (Pessimistic Write Lock)
트랜잭션 시작 시 SELECT ... FOR UPDATE로 users 행에 락을 걸어 동일 사용자의 동시 요청이 직렬화됩니다.
락을 먼저 획득한 요청이 처리되는 동안 나머지 요청은 대기합니다.

3차 방어 — DB Unique Key
purchases 테이블의 (user_id, episode_id) 복합 유니크 인덱스로 앞선 방어를 모두 통과하더라도 DB 레벨에서 최종 중복을 차단합니다.
```

### 방식 비교 및 선택 이유

| 방식 | 장점 | 단점 | 채택 여부 |
|---|---|---|-------|
| Redis 분산 락 | DB 도달 전 차단, 대규모 트래픽에 유리 | Redis 장애 시 락 불가, 별도 인프라 필요 | X     |
| 낙관적 락 | 충돌 적을 때 성능 유리 | 충돌 시 재시도 필요, 결제 도메인에 부적합 | X     |
| 비관적 락 | 트랜잭션 범위 내 정합성 확실히 보장 | 대기 시간 발생, 처리량 감소 가능 | O     |
| DB Unique Key | 구현 단순, DB 레벨 최후 보루 | 중복 시 예외 핸들링 필요 | O     |
| Idempotency Key | 재전송과 신규 요청 구분 가능 | 클라이언트 UUID 생성 필요 | O     |

**비관적 락을 선택한 이유**

결제 도메인은 충돌이 발생했을 때 재시도보다 처음부터 직렬화하는 것이 안전합니다. 낙관적 락은 충돌 감지 후 롤백과 재시도가 필요한데, 코인 차감처럼 금전적 손실이 발생할 수 있는 로직에서 재시도 실패 시 사용자 경험이 나빠질 수 있습니다. 비관적 락은 락을 획득한 순간부터 트랜잭션이 완료될 때까지 다른 요청을 대기시켜 정합성을 가장 확실하게 보장합니다.

**DB Unique Key를 함께 사용한 이유**

비관적 락만으로는 서로 다른 서버 인스턴스에서 동시에 요청이 들어올 경우 락 범위 밖의 경쟁 조건이 발생할 수 있습니다. DB Unique Key는 어떤 경우에도 중복 INSERT 자체를 차단하는 역할을 합니다.

**Idempotency Key를 함께 사용한 이유**

네트워크 오류로 클라이언트가 동일 요청을 재전송하는 경우, 비관적 락과 DB Unique Key만으로는 재전송과 신규 요청을 구분할 수 없습니다. Idempotency Key는 동일 요청의 재전송을 식별해 이미 처리된 요청임을 확인하고 중복 처리를 방지합니다.

**Redis를 선택하지 않은 이유**

Redis 분산 락은 대규모 트래픽 환경에서 유효하지만, Redis 장애 시 락 자체가 불가능해지는 단일 장애 지점이 될 수 있습니다. 별도 인프라 없이 DB만으로 동일한 수준의 정합성을 보장할 수 있는 현재 설계에서 불필요한 복잡도가 추가된다고 생각하여 선택하지 않았습니다.
