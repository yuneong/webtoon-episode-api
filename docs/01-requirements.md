## 요구사항 정리 / 기능 명세

> 1. Users <-> 회차 구매 (Purchase)


1-1. 시나리오 (사용자 관점)
```text
사용자는 웹툰 플랫폼에서 유료 회차를 코인으로 구매할 수 있다.
구매가 완료되면 즉시 해당 회차를 열람할 수 있다.
동일한 회차는 중복 구매되지 않는다.
```

1-2. 기능 명세 (시스템 관점)
- 목적: 사용자는 특정 회차를 코인으로 구매할 수 있어야 하고, 구매 완료 즉시 열람 권한이 부여되어야 한다. 동시에 여러 요청이 들어와도 중복 결제는 절대 발생하지 않아야 한다.
- 주체: 사용자
- 주요 시나리오:
    ```text
    1. 사용자가 구매하려는 회차에 대해 구매 요청을 보낸다.
    2. 시스템은 사용자 및 회차 존재 여부, 중복 구매 여부, 코인 잔액을 확인한다.
    3. 코인을 차감하고 구매 이력과 열람 권한을 생성한 뒤 응답을 반환한다.
    ```
- 시스템 동작 (낙관적):
    ```text
    1. userId, episodeId로 사용자 및 회차 존재 여부 확인
    2. (front idempotencyKey, userId, episodeId) 기준으로 기존 구매 이력 조회 (1차 중복 방어)
    3. 사용자 코인 잔액 확인
    4. 코인 차감
    5. purchase row 생성 (2차 중복 방어)
    6. access row 생성
    7. 구매 ID, 회차 정보, 차감 코인, 구매 일시 반환
    ```
- 예외 처리 (비관적):
    ```text
    1. 존재하지 않는 사용자 ID -> 404
    2. 존재하지 않는 회차 ID -> 404
    3. 이미 구매한 회차 -> 409
    4. 동시 구매 요청으로 DB unique key 충돌 -> 409
    5. 잘못된 코인 형태 (0원, 음수, 숫자가 아닌 형식) -> 400
    6. 필수 요청값이 없을 때 -> 400
    7. 서버 내부 오류 -> 500
    ```
- 체크 리스트
    ```text
    1. 코인 차감 / purchase 생성 / access 생성이 하나의 트랜잭션으로 처리되는지?
    2. 로직 실행 중 실패 시 모든 변경이 롤백되는지?
    3. 동시에 동일한 요청이 들어와도 중복 구매가 발생하지 않는지?
    4. 응답에 구매 ID, 회차 정보, 차감 코인, 구매 일시가 포함되는지?
    5. 구매 완료 즉시 열람 권한이 부여되는지?
    ```

> 2. Users <-> 회차 열람 (Access)


2-1. 시나리오 (사용자 관점)
```text
사용자는 구매한 회차에 자유롭게 접근할 수 있다.
구매하지 않은 회차에 접근하면 열람이 거부된다.
```

2-2. 기능 명세 (시스템 관점)
- 목적: 사용자는 구매한 회차에 한해 콘텐츠를 열람할 수 있어야 한다. 구매 이력이 없는 회차는 접근이 차단되어야 한다.
- 주체: 사용자
- 주요 시나리오:
    ```text
    1. 사용자가 특정 회차에 대해 열람 요청을 보낸다.
    2. 시스템은 사용자 및 회차 존재 여부와 열람 권한을 확인한다.
    3. 권한이 있으면 회차 콘텐츠 정보를 반환하다.
    ```
- 시스템 동작 (낙관적):
    ```text
    1. userId, episodeId로 사용자 및 회차 존재 여부 확인
    2. (userId, episodeId) 기준으로 권한 조회
    3. 회차 정보, 콘텐츠 정보 반환 
    ```
- 예외 처리 (비관적):
    ```text
    1. 존재하지 않는 사용자 ID -> 404
    2. 존재하지 않는 회차 ID -> 404
    3. 열람 권한 없음 -> 403
    ```
- 체크 리스트
    ```text
    1. 구매 이력이 없는 회차는 콘텐츠 URL이 절대 반환되지 않는지?
    2. 응답에 회차 제목, 콘텐츠 URL, 열람 권한 부여 일시가 포함되는지?
    3. 열람 요청은 DB 상태를 변경하지 않는 읽기 전용으로 처리되는지?
    ```
