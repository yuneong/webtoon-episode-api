2. 최초 프롬프트
```text
웹툰 플랫폼에서 사용자가 특정 회차를 구매하면, 해당 회차를 즉시 열람할 수 있도록하는 서버 API를 설계

다수의 사용자가 동일한 회차를 동시에 구매할 수 있고, 동일 회차에 대한 중복 결제는 발생하면 안되고, 서버 장애나 네트워크 오류가 발생해도 결제 정보와 열람 권한은 일관성을 유지해야되는 상황을 전제
위 내용을 바탕으로 요구사항 명세, API 명세(회차구매, 회차열람에 대한 endpoint, request, response, error), 
시퀀스 다이어그램, DB설계(user, episode, purchase, userEpisodeAccess), 중복 결제 방지 전략 작성할거야

중복 결제 방지 전략은 아래 3가지를 모두 사용해서 설계할거야
1. 프론트에서 구매시에 UUID 만들어서 멱등키로 넘겨주기
2. 비관적 락 
3. 디비 유니크 키
```

3. AI결과에 대한 본인 리뷰
```text
AI가 유용하거나 정확했던 부분
- 요구사항 명세, API 명세, 시퀀스 다이어그램 등 문서 구조 잡기
- 마크다운 형식 변환, ERD 초안 작성 등 반복적인 문서 작업
- DB Unique Key의 역할, 트랜잭션 롤백 시나리오 등 기본적인 정합성 개념 정확

AI가 틀렸거나 실무적으로 부적절했던 부분
- 초기에 중복 결제 방지 전략으로 DB Unique Key 단일 방식만 제안, 실제 결제 도메인에서는 동시 요청 직렬화와 네트워크 재전송 케이스까지 고려해야 하므로 단일 방식으로는 부족
- ERD 수정 시 API 명세에 없는 webtoonId를 임의로 추가, 명세 기준으로만 설계해야 하는데 불필요한 컬럼을 추가
- 낙관적 락을 초안으로 제안했으나, 결제 충돌 시 재시도 비용과 사용자 경험을 고려하면 비관적 락이 더 적합하다고 판단

수정 및 보완한 부분
- 비관적 락 + DB Unique Key + Idempotency Key 조합의 3중 방어 구조를 직접 설계하고 그 이유를 명세에 반영
- 시퀀스 다이어그램과 요구사항 명세 간 불일치(코인 잔액 확인 누락, idempotencyKey 흐름 누락 등)를 직접 검토하고 수정
- ERD에서 API 명세 기준으로 필요한 컬럼만 포함하도록 수정

최종 결과에서 본인이 기여한 부분
- 비관적 락 선택 및 3중 방어 구조 설계 결정
- Idempotency Key 도입 및 header 전달 방식 결정
- 요구사항 명세 <-> 시퀀스 다이어그램 <-> ERD 간 일관성 검토
- AI 초안의 오류를 발견하고 명세 기준으로 수정 지시
```
